# 第四轮数据清洗开发总结

## 开发项背景

### 希望解决的问题

第三轮数据清洗后，5150 条记录中仍残留以下问题：

1. **~120 条低精度地址**：地址精度只到县级，DuckDuckGo 对中文偏远遗址搜索效果差
2. **`_geocode_method` 标签不准确**：1679 条标为 `tencent_geocode_gemini`，实际来源是 DeepSeek
3. **26 条父记录坐标为空**
4. **地址字符串缺少行政层级**：如"山东省滕州市官桥镇北辛村"缺少"枣庄市"
5. **全量数据未做最终质量审计**
6. **大型文保单位未拆分**：如"茶马古道"横跨三省 21 个地级市，仅有一条记录

---

## 实现方案

### 关键设计

**多数据源 + 多 Geocoder 组合策略**

本轮最大的改进是引入了多种数据源和地理编码服务的组合：

| 数据源 | 适用场景 | 效果 |
|--------|---------|------|
| 百度百科 API | 有明确词条的文保单位 | 命中率约 60%，位置信息精确 |
| 百度搜索 API | 百科无结果的偏远遗址 | 25 条全部有搜索结果，8 条改善 |
| 维基百科爬取 | 大型遗址拆分（茶马古道） | 提取 182 个坐标点 |

| Geocoder | 特点 | 使用场景 |
|----------|------|---------|
| 腾讯地图 | WGS-84 坐标，reliability 评分 | 前三轮主力，本轮继续用于百科补全 |
| 高德地图 | GCJ-02→WGS-84 转换，level 字段 | 本轮新增，用于搜索补全和重复坐标解决 |

**增量修复工具 `fix_single.py`**

为后续零星修复设计了通用脚本，支持自由组合数据源（百科/搜索）和 Geocoder（高德/腾讯），可单条或批量处理，支持手动指定地址跳过 LLM。

### 剩余 16 条模糊地址

主要是分布范围广的遗址（长城段、烽燧群、墓群等），确实无法定位到单一地点，属于数据本身的固有限制。

### 剩余 12 组重复坐标

多数是紧邻的文保单位（如同一古城内的多处遗址），坐标差异在合理范围内。

## 局限性

1. **16 条模糊地址无法进一步精确**：长城段、烽燧群等大范围遗址的固有限制，只能定位到县级
2. **12 组重复坐标未完全消除**：部分同名/紧邻遗址高德也返回相同坐标
3. **茶马古道拆分粒度为地级市**：实际有 197 个文物点，当前只拆到 21 个地级市代表点
4. **省份验证阻断了少数跨省纠正**：如铁佛寺从临汾纠正到晋城高平被省份验证拦截（百科找到的是对的），后通过百度搜索管道绕过解决

## 后续 TODO

### 1. 剩余模糊地址与重复坐标的攻坚修复

16 条模糊地址和 12 组重复坐标中，部分可通过以下方式继续改善：
- 人工查阅地方志、文物普查报告，获取精确位置
- 对长城段、烽燧群等线性遗址，可选取代表性保存较好的段落作为定位点
- 使用 `fix_single.py` 工具逐条处理

### 2. 系统性全局 review

当前审计仅覆盖"模糊地址"和"重复坐标"两个维度，应进行更全面的排查：
- 坐标离群点检测：同省份记录中坐标明显偏离的异常值
- 地址与坐标一致性校验：地址中的省市县与 geocoding 返回的结构化字段是否匹配
- 名称与描述一致性：检查是否有同名但实际不同的文保单位被错误合并
- 抽样人工核验：随机抽取若干记录在地图上目视验证

### 3. 数据清洗经验沉淀为 skill

四轮数据清洗积累了大量实践经验，应整理为可复用的 skill：
- 地址 geocoding 最佳实践（多数据源选择、省份验证、坐标系转换）
- 大型遗址拆分流程（维基百科解析、按行政区划分组）
- 增量修复工作流（`fix_single.py` 的设计模式）
- 数据质量审计检查清单（模糊地址、重复坐标、离群值等）
