# 需求：Schema 升级——支持一个文保单位多个地址

## 背景

问题：当前 schema 不支持一个全国重点文物保护单位包含多个地址的情况。

典型案例：

- **长城**（第五批，release_id 5-442）：官方公告中列出了 8 个子条目（齐长城遗址、秦长城遗址、燕长城遗址、司马台段、乌龙沟段、雁门关段、清水河段、镇北台），当前做法是把它拆成 8 条完全独立的记录，没有父记录，没有关联关系，技术上不够优雅。
- **唐代帝陵**、**西汉帝陵**等：一个文保单位横跨多个县市，现在只存了一个地理编码点，信息不准确。

## 要解决的问题

1. 在数据模型上优雅地表达"一个文保单位包含多个物理地点"的关系
2. 修复 scraper 中的 off-by-one bug（长城父行未自增 seq，导致第五批约 76 个条目的 release_id 全部偏小 1）
3. 前端详情页展示父子关系

## 实施范围

- Schema 升级：parent_id 自引用外键
- 数据修复：长城 8 个子条目关联到新建的父记录
- Scraper bug 修复：release_id 从 5-442 到 5-517 的条目全部修正
- 前端适配：详情页展示所属文保单位 + 同组分段列表

## 不在本次范围

- 唐代帝陵、西汉帝陵等 ~40 个多地址单位的实际拆分（schema 升级后，数据层面留待第三轮数据清洗处理）
