# 第13轮：筛选优化

## 背景

当前 FilterPanel 有三个体验问题：

1. **类别选项暴露了历史分类**：数据库中存在 10 种类别（含 4 种早期批次的历史分类名），但实际只有 6 种现代分类有意义。用户在下拉框中看到 10 个选项，选择"石窟寺"或"石刻及其他"无法匹配到现代分类的数据。
2. **省份筛选粒度太粗**：只能按 31 个省级行政区筛选，无法精确到市县或任意地址。
3. **年代筛选几乎不可用**：年代字段数据格式不统一（"唐"、"唐至清"、"辽金"、"1841年"），简单的字符串包含匹配会漏掉大量结果。需要 LLM/RAG 方案才能做好，现阶段先移除。

## 需求

### 需求 1：类别 schema 迁移

在数据库 schema 层面解决历史分类问题，而非在前端代码中做映射。

- 新增 `original_category` 字段，存储原始类别值（仅历史分类需要）
- 将 4 种历史分类的 `category` 字段映射到对应的现代分类
- 前端代码中只保留 6 种现代分类，删除所有历史分类的分支

映射关系：
| 历史类别 | → 现代类别 |
|---|---|
| 革命遗址及革命纪念建筑物 | 近现代重要史迹及代表性建筑 |
| 古建筑及历史纪念建筑物 | 古建筑 |
| 石窟寺 | 石窟寺及石刻 |
| 石刻及其他 | 石窟寺及石刻 |

### 需求 2：地址搜索替换省份下拉

将省份下拉框替换为文本搜索输入框，支持用户输入任意地址。

- 第一层（免 API）：在本地数据中匹配 province/city/district/name，匹配到直接筛选 + 定位
- 第二层（调 API）：本地无匹配时调天地图 geocoding API，获取坐标后定位地图（不筛选数据）
- API 耗尽兜底：降级为省份下拉选择

### 需求 3：删除年代筛选

移除年代筛选输入框，从 FilterState 中去掉 era 字段。后续有 LLM/RAG 方案时再重新实现。
