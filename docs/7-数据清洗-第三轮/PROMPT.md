# 第三轮数据清洗需求

## 背景

在第二轮数据清洗和 schema 升级后，`heritage_sites_geocoded.json` 中仍存在三类数据质量问题：

1. **1015 条低精度坐标**：POI 搜索未命中，fallback 到地址编码，坐标仅精确到区县中心（如应县木塔、嘉峪关等知名遗址坐标落在市区中心）

2. **多地址条目未拆分**：唐代帝陵、西汉帝陵、京杭大运河等约 40 个文保单位在官方公告中明确包含多个物理地点，但数据中仍是单条记录。上一轮只处理了长城（8 个子条目），其余待处理。

3. **重复坐标**：多条记录共享完全相同的经纬度，大概率是 POI 搜索错误（如"十二连城"在内蒙古、山西、陕西各有一处，三条记录却共用一个坐标）。

## 约束

高德地图 API 本月配额已耗尽，不能使用高德。本轮采用：

- **腾讯地图 Web 服务 API**：与高德同为 GCJ-02 坐标系，无需坐标转换，个人版每日 10,000 次免费额度
- **Gemini Deep Research**：辅助研究更准确的地址和 POI 名称

## 核心需求

1. **数据诊断**：全面扫描并分类所有数据问题（低精度坐标、多地址候选、重复坐标、省份不匹配）

2. **多地址拆分**：
   - 借助 Gemini 判断哪些条目需要拆分，并提供子条目信息
   - 脚本自动完成拆分操作
   - 只做拆分，不做合并（不同批次的同名条目保持独立）

3. **坐标修正**：
   - 借助 Gemini 提供更准确的地址/POI 名称
   - 用腾讯地图 API 重新 geocoding
   - 关键改进：加入省份验证，拒绝跨省错误匹配

4. **验证与收尾**：重新 seed 数据库，前端验证

## 工具和方法

- Gemini：传入 JSON 附件，一次处理全量记录，返回结构化 JSON
- 腾讯地图：POI 搜索 + 地址编码双策略，含省份验证
- 人工：对仍无法自动解决的记录手动填写坐标
